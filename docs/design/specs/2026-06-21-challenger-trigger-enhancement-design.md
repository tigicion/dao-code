# 挑战者触发场景完善 — 设计

日期:2026-06-21
状态:已批准设计,待写实现计划

## 背景与动机

反思层有两个角色(`src/agent/reflect_prompts.ts`):**挑战者**(卡住时怀疑性复核)、**纠偏者**(长任务周期性复述目标、查 scope 蔓延)。本次只动**挑战者**的触发。

实测发现挑战者有两个盲点:

1. **"假进展"逃逸**:`assessTurn`(`src/agent/turn_health.ts`)里 `stuck = toolFailures>0 && !progressed`,而 `progressed` 仅判"本轮调没调过写/改类工具"(`loop.ts` 的 `PROGRESS_TOOLS`)。于是 agent **只要改了文件就算"有进展"**、`failureStreak` 被复位——即便每轮构建/测试照样失败。"乱改一通、错误还在"被误判成在推进。
   - 注:**改文件 + 同一错误复发**已被 `repeatedErr`(同错签名计数,与 progressed 无关)覆盖。真正逃逸的是**改文件 + 每轮换一个新错误**:`progressed=true` 让 failureStreak 复位、`errSig` 每轮变又让 repeatedErr 复位 → 两条都触发不了,看着"很忙"实则空转。

2. **"用户反复申诉"无人听**:挑战者只盯**工具失败**,不盯**用户反馈**。实测让 dao 找一个不存在的 bug(画面不显示):它读代码、xcodebuild 全成功(零失败),原地绕圈猜测 14 轮到上限——挑战者全程不触发。而真实场景里,**用户回过头说"还是不行/依然空白"本身就是"你判断错了"的最强信号**,当前完全没接进触发逻辑。

核心洞察:**"进展"的 ground truth 不是"碰没碰文件"、甚至不是"构建过没过",而是"用户的问题解决没有"。用户反复重提同一问题 = 权威的"未解决"信号。**

## 范围

- 改进挑战者的**两条触发路径**(下述路径①新增、路径②修正)。
- **不改**纠偏者(已是每 N 轮的节奏触发,本次不动)。
- **不做**纯时间型"长时间零推进"兜底(易误伤长探查任务,YAGNI)。

## 设计

### 路径② — 修正"假进展"(任务中途,改现有)

`assessTurn` 的卡住判定不再被"改了文件"赦免:**本轮有工具失败,就是没推进**,无论是否动过文件。

- 现:`stuck = outcome.toolFailures > 0 && !outcome.progressed`
- 改:`stuck = outcome.toolFailures > 0`

效果:"改文件 + 每轮换新错误"的空转会让 `failureStreak` 正常累积到阈值 → 触发挑战者。`progressed` 字段在卡住判定中不再使用(仍可保留给 `loop.ts` 的 noProgress advisor;那是独立的温和提醒,不在本次范围)。

单个偶发失败不会误触发——`failureStreakTrip`(默认 3)提供迟滞。

### 路径① — 用户反复申诉触发(交互式,新增)

**两段式:免费相似度门 → pro 挑战者判断。**

1. **相似度门(零模型,确定性)**:交互式会话里收到一条新用户消息时,用 `textSimilarity`(`src/text/similarity.ts`,字符二元组 Jaccard,CJK 鲁棒)把它与**本会话之前的各条用户消息**比对,取最高分。
   - 最高分 ≥ 阈值(默认 `0.5`,可 env 调)→ 判为"用户在重提同一件事" → 进第 2 步。
   - 否则(新任务/新需求)→ 不触发,静默。
   - 阈值偏召回:宁可多放一点进挑战者(它会否掉碰巧措辞像的新问题),也不漏真申诉。

2. **pro 挑战者判断(异步,不阻塞主流程)**:相似度门命中才 fork 一次挑战者(复用主前缀缓存,边际便宜)。挑战者**自己**判这是不是"原问题没解决的申诉",以及 agent 是否在空转/前提是否错;若其实是新任务/误报,它回"在轨,继续",不打扰。

   **执行模型(关键)**:挑战者**异步起跑、不挡主流程**——用户消息一进来,agent 立刻开干,挑战者在旁边跑。结论就绪后走**回合边界注入通道**(`loop.ts` 现有 `drainPending`,每个工具回合边界 drain 一次):
   - **本回合内尽量接住**:若挑战者在 agent 还在跑工具回合时返回,结论在**下一个工具回合边界**被注入,agent **当轮就看到**、可立即调整。
   - **赶不上就晚一回合**:若 agent 本回合已结束(停止调工具)挑战者才返回,结论在**下一个用户回合**注入。延迟一回合可接受。
   - `reflect()` 在入口处 fork 时 `[...session.messages]` 即时快照,异步运行不受主循环后续消息写入影响(读侧无竞态);写侧只经回合边界 drain(安全注入点)。

**为何这样设计**:DAO 交互循环里 agent 每次都先跑完再交还控制权,所以"agent 刚结束"对几乎每条用户消息都成立——若以此为门会**每轮都 fork**(成本可控但**每轮加延迟**,体感差)。"重复"才是 phantom-bug / 反复申诉的真信号,而重复可用文本相似度**免费**检测,把昂贵的 pro 调用收敛到真正需要的时刻。

### 挑战者 prompt 增强

`CHALLENGER_PROMPT` 现仅围绕"卡住"。增补一段,使其在两条路径下都适用:让它复述"用户最初要什么 / 我声称做了什么",并明确自查:
- 我的"修复"是否真的落地解决了问题,还是只是改了文件、换了个错误?
- 用户是否在重复表达"还没好"?若是,**停止再叠加修复**,质疑诊断/前提:bug 是否真在我以为的地方?是否该从头复现真实症状,或向用户要具体复现步骤?
- 若一切正常、这只是新任务 → 直接回"在轨,继续",不要硬找茬。

保持"只评估、不干活、输出精简结论"的现有约定。

## 架构与数据流

- **路径②**:纯改 `assessTurn`(`turn_health.ts`)一处布尔式。无新状态。
- **路径①**:
  - 新增一个纯函数(放 `turn_health.ts` 或新建小模块),签名约 `isRepeatComplaint(newMsg: string, priorUserMsgs: string[], threshold: number): boolean`,内部用 `textSimilarity` 取与历史各条的最高分比阈值。**纯函数、可单测**。
  - **触发点在用户消息入口、agent 开跑之前**(`src/index.ts`:REPL 路径 `runOneTurn` 起点 / Ink 路径对应处;一次性 argv 路径不接,延续 `reflect: argvPrompt ? undefined` 的现状)。
  - 命中 → **不 await**:`void reflect("challenger")` 异步起跑(`reflect()` 已在 `index.ts` 作用域内、pro/主模型 fork + 前缀缓存),其 resolve 回调把结论推入一个**待注入队列**;不阻塞 `runTurn` 立即开跑。
  - 主 `runTurn` 接一个 `drainPending`(现有 hook,`loop.ts:160`),每个工具回合边界 drain 该队列 → 有结论就作为 advisory 注入(建议 `system` 角色、`[审视者·参考]` 前缀,而非现有的 `[追加指令]` user 角色)。本回合接住即当轮生效;赶不上则 REPL 在下一个用户回合 drain。
  - `reflect()` fork 时 `[...session.messages]` 即时快照 → 异步运行不受后续写入影响(读侧无竞态);写侧只经回合边界 drain(唯一安全注入点)。
  - 历史用户消息从 `session.messages`(role==="user")取;新消息即将入会话,比对的是**它之前**的那些。
  - 需要的接线:给主 `runTurn`(`runOneTurn`)接上 `drainPending`(当前主循环未接);并让回合边界 drain 支持注入 `system` advisory(非仅 user)。

## 配置与可观测

- `DAO_CHALLENGE_REPEAT_SIM`:相似度门阈值,默认 `0.5`;设 `0` 关闭路径①。
- `DAO_REFLECT=0`:仍是反思层总开关(挑战者+纠偏者全关),不变。
- 触发时沿用现有 `events.notice("[反思:审视当前进展…]")`,用户可见。

## 错误处理

- 相似度门是纯计算,不会抛;无历史用户消息时返回 false(首条消息不触发)。
- 挑战者 fork 失败(`reflect()` 已 try/catch 返回 null)→ 静默跳过,绝不阻断主流程。

## 测试

- **路径②**:`turn_health.test.ts` 加用例——"改了文件但本轮仍有工具失败 → 仍判 stuck、failureStreak 累积"(现行为下会被 progressed 赦免,新行为不赦免)。
- **路径①相似度门**:纯函数单测——重提同一问题(中英、含改写)→ true;全新任务 → false;无历史 → false;阈值边界。
- **挑战者 prompt**:快照/包含断言,确认新增自查项在位。
- **异步注入**:待注入队列纯逻辑单测——结论入队后,回合边界 drain 取出为 `system` advisory(`[审视者·参考]`);未就绪时 drain 返回空、不阻塞。`loop.ts` drain 支持 system 角色的注入(现仅 user)。
- 全量 `vitest run` 无回归。

## 不做(YAGNI)

- 纯时间型"零推进"兜底。
- 改写型重复的语义召回(纯词面相似 + pro 否误报已够;不引第二个模型做门控)。
- 纠偏者改动。
- 一次性/eval 路径接入反思(刻意保持测量干净)。
