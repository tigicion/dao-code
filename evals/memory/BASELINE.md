# 记忆效果评测 — 首轮真实基线与发现(2026-07-01)

在 14 个真实(脱敏)用例上跑了首轮基线(8 提取 + 6 召回,`EVAL_JUDGE_K=1` directional)。素材取自真实开发会话/记忆库,金标由人审定稿。结论如下。**报告本身(`report.md`/`runs/`)不入仓;本文件沉淀结论。**

## 一、召回效果(可信)

当前注入 = **「active 记忆按 importance×衰减 全量注入,无语境检索」**。6 个真实用例一致:

| 指标 | 结果 |
|---|---|
| 价值 Recall | **1.00**(该进的全进了) |
| 价值 Precision | 0.57–0.67(干扰记忆也一起注入) |
| stale 泄漏 | 0 |
| 相关性缺口(对人工金标) | 0.00 |
| judge-人工相关性一致度 F1 | 0.86–1.00 |

**结论**:小库(≤cap 50)上"全塞"→ 召回满分但**零区分度**,精确率被干扰项拖到 ~0.6;相关性缺口 0 是"全塞"的副产物而非智能。真正的召回缺口只有当库 > cap 时才暴露。**这量化了"无检索"短板 → 是 scope-tagging(语境相关性检索)那条 backlog 的直接动机。**

## 二、提取效果(质量好,但"分数"不可靠)

- ✅ **reflect 未坏,抽取质量高**:judge 对抽出记忆的四维质量打分稳定在 **0.85–1.00**;**用户画像确实能被抽出**(如"用户长期构建儿童 iPad 应用"、"进度不入记忆并提及 2 岁孩子")。
- ⚠️ **单采样极不确定**:同一 case 两次跑,抽出记忆数在 0↔3 跳动,per-case factRecall/profileRecall 大幅摆动。**单次抽取的 per-case 分数不能当基线。**(spec 已标 `EVAL_EXTRACT_K` 预留;本轮实证其必要性。)
- ⚠️ **固定金标是"一份"答案而非唯一**:reflect 常抽出**另一批同样合法的耐久事实**(如"中文 iOS 开发者/偏好根因诊断"),被固定金标覆盖率算作 miss,压低 factRecall。

**结论**:提取质量本身是好的、画像抽得出;但用"固定金标覆盖率 + 单采样"给它打绝对分,数字既噪又严,只能作方向性参考,不能当结论性评分。

## 三、这一轮抓到并修掉的 eval 自身缺陷

真实跑批的价值,一半在于暴露评测自身的坑:
1. **redactEvents 漏脱敏 `toolCalls[].args`**(会把路径/专名泄漏进进仓 fixture)→ 已修 + 测试(`bbabf1b`)。
2. **召回 fixture 的 `source:(redacted)` 被 validateMemory 误判 stale** → 记忆被错误排除出注入(slide 5 条全中招 → 假的 0 分)。去掉 source 后召回数字才可信(`b9e244f`)。
3. **提取用了 `fork:false`(关思考)** ≠ 线上 `fork:true` 带推理 → 改为对齐线上(`9224847`)。
4. **谄媚默认值**:无画像金标/无抽出时"画像召回/精确率"曾显 1.0 → 改显 **N/A**(`9224847`)。

## 四、已知局限与未来方向(要更可信的提取基线需要)

- **多采样提取**:实现 `EVAL_EXTRACT_K`,每 case 跑 reflect K 次取中位+方差,平掉单采样噪声。
- **更公道的提取指标**:从"是否抽中金标那几条"转向"抽出的东西有多少是耐久/画像/在题"(靠 quality + 软召回),不惩罚 reflect 抽出另一批合法事实。
- **输入保真度**:eval 把 events.jsonl 经 toMessages 重建成 messages 喂 reflect,工具结果被截断、工具调用只留名——是线上输入的有损近似;更保真的重建会让提取更接近线上。
- **judge K=3**:本轮 K=1 directional;收紧用 K=3 取中位。
